# Implementation Plan: Menu Management

**Branch**: `001-menu-management` | **Date**: 2026-05-31 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `/specs/001-menu-management/spec.md`

## Summary

ServTable needs a complete dish management system allowing restaurant managers to create, read, update, and delete menu items with support for multiple price tiers, image uploads, availability toggling, and buffet position organization. The frontend leverages React with TypeScript, React Query for server state, and React Hook Form for form management, following a strict service layer architecture where all API calls route through dedicated services and business logic lives in custom hooks.

## Technical Context

**Language/Version**: TypeScript 5.x with React 18.x

**Primary Dependencies**: React, React Query (TanStack Query), React Hook Form, Zod, Vite, Tailwind CSS

**Storage**: Backend REST API (no frontend-side storage; all state managed server-side)

**Testing**: Jest + React Testing Library (unit/integration), manual UI testing against existing components

**Target Platform**: Web browser (responsive, desktop/tablet focus)

**Project Type**: React web frontend

**Performance Goals**: <500ms API response time for dish CRUD operations, <1s image upload completion

**Constraints**: Must reuse existing UI components; no redesign; strict TypeScript typing (no implicit `any`)

**Scale/Scope**: Menu management for single buffet/restaurant (hundreds of dishes, dozens of concurrent managers)

## Design Decisions

### 1. Service Layer Architecture
- **Decision**: Centralize all API calls in `src/services/dishes/dishService.ts`
- **Rationale**: Enables testing, mocking, and swapping implementations. Separates business logic from UI.
- **Implementation**: Service exports query/mutation functions and Zod schemas; hooks wrap these with React Query

### 2. Custom Hooks for State Management
- **Decision**: Create specialized hooks (`useDishes`, `useCreateDish`, etc.) wrapping React Query
- **Rationale**: Encapsulates query logic, error handling, and side effects. Components remain simple and focused.
- **Implementation**: Each hook handles a single responsibility; hooks compose `dishService` functions

### 3. Validation with Zod
- **Decision**: Define Zod schemas for all dish data; validate at service boundaries
- **Rationale**: Ensures runtime type safety; catches API response mismatches; provides automatic TypeScript types
- **Implementation**: Schemas cover Dish entity, Create/Update/Delete payloads, list responses

### 4. Image Upload Strategy
- **Decision**: Upload images as multipart form data via dedicated endpoint; store reference in dish record
- **Rationale**: Separates file storage from metadata; allows image replacement by overwriting
- **Implementation**: Service handles multipart serialization; hooks abstract upload complexity

### 5. React Hook Form for Dish Forms
- **Decision**: All dish CRUD forms use React Hook Form + Zod validation
- **Rationale**: Type-safe form handling, integration with Zod, minimal re-renders
- **Implementation**: Create form schemas derived from validation schemas; hooks provide pre-filled data for updates

## API Contract Summary

### REST Endpoints (Backend)

| Operation | Method | Endpoint | Payload | Response |
|-----------|--------|----------|---------|----------|
| List dishes | GET | `/api/buffets/{buffetId}/dishes` | — | `{ dishes: Dish[] }` |
| Get dish | GET | `/api/buffets/{buffetId}/dishes/{dishId}` | — | `Dish` |
| Create dish | POST | `/api/buffets/{buffetId}/dishes` | `CreateDishInput` | `Dish` |
| Update dish | PUT | `/api/buffets/{buffetId}/dishes/{dishId}` | `UpdateDishInput` | `Dish` |
| Delete dish | DELETE | `/api/buffets/{buffetId}/dishes/{dishId}` | — | `{ success: true }` |
| Upload image | POST | `/api/buffets/{buffetId}/dishes/{dishId}/image` | multipart/form-data | `{ imageUrl: string }` |

### Data Models

**Dish Entity**:
```typescript
{
  id: string;
  name: string;
  description?: string;
  ingredients?: string;
  smallPrice: number;      // > 0
  mediumPrice: number;     // > 0
  largePrice: number;      // > 0
  imageUrl?: string;
  buffetPosition: number;  // > 0, unique per buffet
  available: boolean;      // defaults to true
  createdAt: ISO8601;
  updatedAt: ISO8601;
}
```
